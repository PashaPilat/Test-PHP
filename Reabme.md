### Функциональные требования

>1. Создать каталог товаров с категориями, связать их между собой.  
Для примера, можно взять данные с [demo.solomono.net](http://demo.solomono.net)  
>2. Создать в базе 10-20 товаров (товар должен иметь, дату, цену, назвавние товара), и 3-5 категорий
>3. Вывести страницу.  
>4. При клике на категорию, список товаров обновляется **без перезагрузки страницы**

>Страница должна содержать:
>- список категорий слева
>- список товаров справа
>- количество товаров в категории
>- сортировку (по цене, алфавиту, новые), по дефолту дешовые
>- AJAX обновление

>У каждого товара:

>- кнопка **Купить**
>- открывается **модальное окно Bootstrap**

### ⚙ Архитектура SQL Builder

> **SQL Builder** — это инструмент для построения SQL‑запросов через цепочку методов (*fluent interface*).  
> Он позволяет описывать запросы в коде так, чтобы они были читаемыми, гибкими и легко поддерживаемыми.

---

### Почему это лучше

- **Читаемость**: запросы выглядят как последовательность шагов, а не как строка SQL.  
- **Безопасность**: параметры передаются отдельно, что снижает риск SQL‑инъекций.  
- **Расширяемость**: легко добавлять новые методы (`join`, `groupBy`, `limit`).  
- **Унификация**: один стиль написания запросов для всей команды.  

---

### Builder

SQL Builder реализует подход **fluent interface**, позволяющий строить запросы цепочкой методов.  
Это делает код читаемым и гибким, а также упрощает сопровождение и расширение.  
Внутренние основные методы билдера:

- `select(columns)` — выборка колонок
- `from(table)` — указание таблицы
- `where(column, operator, value)` — фильтрация
- `join(table, leftColumn, rightColumn, type)` — объединение таблиц
- `groupBy(column)` — группировка
- `orderBy(column, direction)` — сортировка
- `limit(count)` — ограничение количества строк
- `all()` — выполнение запроса и получение результата

Пример:
```php
    DB::select()
        ->from()
        ->where()
        ->orderBy()
        ->limit()
```
---

### Where

Модуль фильтрации данных.  
Позволяет накладывать условия на выборку в формате:

    WHERE column operator value

Поддерживаются различные операторы (=, >, <, LIKE, IN), а также логические связки (AND, OR).  
Можно комбинировать условия для сложных фильтров.

---

### Join

Механизм объединения таблиц.  
Поддерживаются типы соединений:

- INNER JOIN — выборка только совпадающих записей
- LEFT JOIN — выборка всех записей из левой таблицы, даже если нет совпадений
- RIGHT JOIN — аналогично, но для правой таблицы
- FULL JOIN — объединение всех записей обеих таблиц

---

### GroupBy

Позволяет группировать данные по выбранным колонкам.  
Используется для агрегации (например, подсчёта количества товаров в категории).  
Поддерживает использование агрегатных функций (`COUNT`, `SUM`, `AVG`).

---

### OrderBy

Сортировка данных по указанным полям.  
Поддерживаются направления:

- ASC — по возрастанию
- DESC — по убыванию

Можно комбинировать несколько полей для сложной сортировки.

---

### Limit

Ограничение количества возвращаемых строк.  
Применяется для пагинации и оптимизации выборок.  
Поддерживает смещение (`OFFSET`) для постраничного вывода.

---

### Enum для типизации

Для предотвращения ошибок в SQL используются перечисления:
```
    JoinType
    JoinOperator
    WhereOperator
    Boolean
    OrderDirection
```    

Это гарантирует корректность передаваемых параметров и снижает риск опечаток.

---

### 📖 Примеры использования

Простая выборка:

    $rows = DB::select('id', 'name')
        ->from('products')
        ->where('status', 'active')
        ->orderBy('price', OrderDirection::ASC)
        ->all();

SQL:

    SELECT id, name
    FROM products
    WHERE status = ?
    ORDER BY price ASC

---

JOIN пример:

    $rows = DB::select('*')
        ->from('categories')
        ->join(
            'products',
            'categories.id',
            'products.category_id'
        )
        ->all();

SQL:

    SELECT *
    FROM categories
    INNER JOIN products
    ON categories.id = products.category_id

### Структура миграций

> **Миграции** — это механизм пошагового создания и изменения структуры базы данных.  
> Они позволяют воспроизводимо поднимать схему, откатывать изменения и гарантировать одинаковое состояние у всех разработчиков.

Файлы миграций хранятся в каталоге:

    migrations/

Пример структуры:

    001_create_categories.php
    002_create_products.php
    003_create_product_category.php
    004_create_attributes.php
    005_create_product_attributes.php
    006_create_table_counter.php
    007_insert_table_data.php

Каждый файл отвечает за создание или модификацию конкретной таблицы/сущности.  
Нумерация обеспечивает порядок выполнения.

---

### Запуск миграций

> Для применения миграций используется команда:

    php migration.php up

Она последовательно выполняет все доступные миграции и создаёт необходимые таблицы.

---

### Откат миграций

> Для возврата базы данных в предыдущее состояние:

    php migration.php down

Команда удаляет созданные таблицы и возвращает схему к исходному виду.  
Это удобно для тестирования и отладки.

### Класс CatalogController

В `CatalogController` реализован паттерн **Controller** из MVC.  
Метод `index()` собирает данные из сервисов (`CategoryService`, `ProductService`, `MenuService`) и передаёт их в представление.  
Такой подход позволил разделить ответственность: контроллер управляет потоком данных, сервисы — бизнес‑логикой, а представления — отображением.  
Использование композиции сервисов делает контроллер лёгким и тестируемым.

---

### Класс ApiController

В `ApiController` реализован паттерн **Controller**, но для AJAX‑взаимодействия.  
Метод `getProduct()` получает данные через `ProductService` и возвращает JSON‑ответ через `Response`.  
Здесь применён паттерн **Facade**: `Response::json()` скрывает детали формирования ответа.  
Такой подход позволил унифицировать API‑слой и сделать его удобным для фронтенда.

---

### Класс ErrorHandler

`ErrorHandler` — статический обработчик ошибок.  
Использован паттерн **Singleton** (статические методы и состояние) и **Observer** (подписка на события ошибок через `set_exception_handler`, `set_error_handler`, `register_shutdown_function`).  
Методы `handleException`, `handleExceptionDB`, `handleError`, `handleShutdown` буферизуют ошибки и управляют их выводом.  
Такой подход позволил централизовать обработку ошибок, разделить режимы (CLI/WEB, debug/production) и гарантировать единый формат логирования.

---

### Класс CategoryService

`CategoryService` инкапсулирует бизнес‑логику работы с категориями.  
Метод `getBySlug()` использует `DB::select()` и паттерн **Repository** для выборки данных.  
Создание объекта `Category` и вызов `fill()` выражает паттерн **Data Mapper**: данные из БД преобразуются в модель.  
Такой подход позволил отделить SQL‑логику от бизнес‑логики и сделать код расширяемым.

---

### Класс DB

`DB` — фасад для работы с базой данных.  
Методы `select`, `insert`, `update`, `delete` возвращают билдеры, реализуя паттерн **Builder**.  
Использован паттерн **Facade**: единая точка доступа к подсистемам работы с SQL.  
Метод `toSql()` и объект `ToSql` обеспечивают воспроизводимость запросов.  
Такой подход позволил сделать API базы читаемым, гибким и безопасным.

---

### Класс Blueprint

`Blueprint` описывает структуру таблиц для миграций.  
Методы `id()`, `integer()` и другие инкапсулируют логику генерации SQL‑определений.  
Использован паттерн **Builder**: таблица описывается декларативно, а затем преобразуется в SQL.  
Такой подход позволил вынести логику создания схемы в отдельный слой и сделать миграции удобными для сопровождения.

---

### Контракт FromContract

`FromContract` задаёт контракт для части SQL‑билдера.  
Методы `set()` и `get()` определяют минимальный набор обязанностей для работы с таблицей.  
Здесь выражен принцип **Interface Segregation** и паттерн **Contract/Interface**.  
Такой подход позволил разделить обязанности между разными частями билдера и сделать архитектуру гибкой и расширяемой.

---


### Итоговое описание архитектуры

Архитектура проекта построена на сочетании **MVC** и сервисного слоя.  
Контроллеры управляют потоком данных, сервисы инкапсулируют бизнес‑логику, а фасад `DB` и билдеры обеспечивают удобную работу с SQL.  
Миграции и `Blueprint` дают воспроизводимую схему базы, а `ErrorHandler` централизует обработку ошибок.  
В итоге проект выражает принципы **SOLID**, использует ключевые паттерны (**Facade**, **Builder**, **Repository**, **Observer**, **Data Mapper**) и обеспечивает модульность, расширяемость и удобство сопровождения.

### Таблица categories

Назначение: хранение категорий товаров в виде дерева.  
Ключевые поля:  
- `id` — уникальный идентификатор категории.  
- `name`, `slug`, `path` — название, алиас и путь от корня.  
- `parent_id` — родительская категория (FK → categories.id).  
- `description`, `icon`, `status`.  
Индексы: `status`, `slug`, `parent_id`.  
Особенности: триггеры для временных меток, FK для parent_id.  
Роль: основа иерархии каталога.

---

### Таблица products

Назначение: хранение товаров.  
Ключевые поля:  
- `id`, `name`, `slug`.  
- `description`, `image`.  
- `price`, `old_price`, `status`.  
- Метрики: `views`, `purchases`, `likes`, `favorites`, `rating`, `rating_count`.  
Индексы: `slug`, `status`, `price`, `old_price`, `created_at`, `views`, `purchases`, `likes`, `favorites`, `rating`, `rating_count`.  
Особенности: триггеры для временных меток.  
Роль: основная сущность каталога.

---

### Таблица product_category

Назначение: связь многие‑ко‑многим между товарами и категориями.  
Ключевые поля:  
- `product_id` (FK → products.id).  
- `category_id` (FK → categories.id).  
Индексы: уникальный составной ключ `product_id + category_id`.  
Роль: обеспечивает гибкую привязку товаров к категориям.

---

### Таблица attributes

Назначение: хранение атрибутов товаров.  
Ключевые поля:  
- `id`, `name`, `slug`.  
- `description`, `type`, `status`.  
Индексы: `slug`, `status`.  
Особенности: триггеры для временных меток.  
Роль: определяет характеристики товаров.

---

### Таблица product_attributes

Назначение: хранение значений атрибутов для товаров.  
Ключевые поля:  
- `id`.  
- `product_id` (FK → products.id).  
- `attribute_id` (FK → attributes.id).  
- `value`, `description`, `status`.  
Индексы: `product_id`, `attribute_id`, `status`.  
Роль: связывает товары с их конкретными характеристиками.

---

### Таблица counters

Назначение: агрегирующая таблица для подсчётов и диапазонов цен.  
Ключевые поля:  
- `entity_type` — тип сущности (category, attribute).  
- `entity_id`, `category_id`.  
- Счётчики: `children_count`, `products_count`, `attributes_count`.  
- Диапазон цен: `min_price`, `max_price`.  
- `values_json` — JSON со значениями атрибутов.  
Индексы: уникальный составной ключ `entity_type + entity_id + category_id`, индекс `entity_type + category_id`.  
Особенности: триггеры для автоматического обновления счётчиков при изменении категорий, товаров и атрибутов.  
Роль: обеспечивает консистентность данных и быстрый доступ к агрегированным значениям.

---

### Представления

- `categories_with_count` — категории с подсчётами (children_count, products_count, attributes_count, min_price, max_price).  
- `attributes_with_count` — атрибуты с подсчётами (products_count, attributes_count, values_json).  
Роль: упрощают выборку данных для фронтенда и сервисов.

---

### Итоговое описание

Структура базы данных построена вокруг сущностей **категории**, **товары** и **атрибуты**, связанных через промежуточные таблицы.  
Таблица `counters` и триггеры обеспечивают автоматическое обновление агрегированных данных, что делает систему устойчивой и удобной для фильтрации и аналитики.  
Представления (`categories_with_count`, `attributes_with_count`) дают готовые выборки для сервисов и фронтенда, снижая нагрузку на бизнес‑логику.  
В итоге архитектура базы данных обеспечивает гибкость, консистентность и высокую производительность при работе с каталогом.

### Файл app.js

Назначение: точка входа фронтенд‑логики.  
Содержит: подключение Bootstrap, jQuery, noUiSlider, глобальное объявление `$` и `bootstrap`.  
Создаёт объект `PlatPCore` и вызывает `init()`.  
Роль: инициализация всего клиентского приложения.

---

### Класс Api

Назначение: слой взаимодействия с сервером через AJAX.  
Методы:  
- `request(url, data)` — базовый метод для GET‑запросов.  
- `getProducts(params)` — загрузка товаров.  
- `getCategories(params)` — загрузка категорий.  
- `updateProductCart(params)` — обновление корзины.  
Паттерн: **Facade** — единая точка доступа к API.  
Роль: абстрагирует работу с сервером и упрощает вызовы.

---

### Класс PlatPCore

Назначение: ядро фронтенд‑приложения.  
Содержит: инициализацию `Api`, `HistoryManager`, `Catalog`, `Sidebar`, `Filters`, `Cart`.  
Метод `init()` запускает все модули.  
Паттерн: **Mediator** — связывает разные части приложения.  
Роль: обеспечивает согласованную работу модулей и централизованное управление.

---

### Класс HistoryManager

Назначение: управление URL каталога через History API.  
Методы:  
- `init()` — подписка на событие `popstate`.  
- `push(state)` — обновление URL.  
- `buildUrl(state)` — построение URL из состояния каталога.  
Роль: синхронизирует состояние каталога и адресную строку.

---

### Класс Cart

Назначение: управление корзиной товаров.  
Содержит: хранение состояния корзины в `localStorage`, модальное окно Bootstrap.  
Методы:  
- `refreshCart()` — обновление содержимого корзины.  
- `bindOpen()`, `bindBuy()`, `bindQtyChange()`, `bindRemove()`, `bindClear()` — обработка действий пользователя.  
Роль: обеспечивает работу корзины, синхронизацию с сервером и отображение в интерфейсе.

---

### Класс Catalog

Назначение: управление каталогом товаров.  
Содержит: загрузку товаров и категорий, работу с состоянием, обработку пагинации, сортировки и фильтров.  
Методы:  
- `init()` — инициализация каталога.  
- `loadCategory()` — загрузка категории или всех товаров.  
- `restoreState()` — восстановление состояния из истории.  
- `setPage()`, `setSort()`, `setFilter()` — управление состоянием.  
Роль: основной модуль каталога, связывает API, HistoryManager и Filters.

---

### Класс CatalogState

Назначение: хранение состояния каталога.  
Содержит: `category`, `page`, `sort`, `filters`.  
Методы: `setCategory()`, `setPage()`, `setSort()`, `setFilter()`, `removeFilter()`, `getParams()`, `setState()`.  
Роль: обеспечивает единый источник состояния для каталога и HistoryManager.

---

### Класс Filters

Назначение: управление фильтрами каталога.  
Содержит: инициализацию ползунка цены через noUiSlider, обработку чекбоксов и секций фильтров.  
Методы:  
- `initPriceSlider()` — настройка диапазона цен.  
- `bindCheckboxFilters()` — обработка чекбоксов.  
- `bindToggleSections()` — скрытие/открытие секций фильтров.  
Роль: обеспечивает гибкую фильтрацию товаров.

---

### Класс Product

Назначение: управление страницей товара.  
Методы:  
- `init()` — инициализация, обработка кликов по категориям.  
- `loadProducts()` — загрузка товаров по slug.  
Роль: модуль для работы с отдельной страницей товара.

---

### Класс Sidebar

Назначение: управление боковой панелью каталога.  
Методы:  
- `init()` — инициализация.  
- `bindToggle()`, `bindCategoryClick()` — обработка действий пользователя.  
Роль: обеспечивает работу бокового меню и взаимодействие с категориями.

---

### Итоговое описание

JS‑архитектура проекта построена вокруг ядра `PlatPCore`, которое связывает все модули: `Api`, `Catalog`, `Cart`, `Filters`, `Sidebar`, `HistoryManager`.  
Каждый модуль отвечает за свою область: API — взаимодействие с сервером, Catalog — управление товарами и категориями, Cart — корзина, Filters — фильтрация, Sidebar — боковое меню, HistoryManager — синхронизация URL.  
Состояние каталога хранится в `CatalogState`, что обеспечивает воспроизводимость и согласованность работы.  
В итоге архитектура фронтенда модульная, расширяемая и полностью синхронизирована с сервером и интерфейсом.

### Бэкенд

- Не реализована полноценная система авторизации и ролей (админ, пользователь).  
- Логика сервисов (`ProductService`, `CategoryService`) пока ограничена базовыми выборками, отсутствует расширенный функционал (например, массовые операции, кэширование).  
- Триггеры и счётчики работают, но нет тестов для проверки корректности при сложных сценариях (удаление, массовое обновление).  
- Ошибки логируются, но нет гибкой системы уведомлений (например, email или Slack‑интеграция).

---

### Фронтенд

- Модули JS покрывают каталог, корзину, фильтры, но нет полноценного модуля для страницы товара (реализован частично).  
- Нет единой системы уведомлений и UI‑компонентов (сейчас используется `notify`, но без централизованного менеджера).  
- Фильтры работают, но не все типы атрибутов поддержаны (например, boolean).  
- История (`HistoryManager`) синхронизирует URL, но не восстанавливает все состояния фильтров при сложных комбинациях.  
- Нет unit‑тестов для JS‑модулей, что усложняет отладку.

---

### База данных

- Структура таблиц и триггеров реализована, но отсутствует система миграций для тестовых данных (сидирование ограничено root‑категорией).  
- Таблица `counters` покрывает категории и атрибуты, но не учитывает дополнительные метрики (например, популярность фильтров).  
- Представления (`categories_with_count`, `attributes_with_count`) есть, но можно расширить для аналитики (например, топ‑товары по просмотрам).  
- Нет отдельной таблицы для пользователей и заказов — проект пока ограничен каталогом и корзиной.

---

### Документация

- README содержит архитектуру, структуру проекта и базы, но нет раздела по деплою (Docker, CI/CD).  
- Не описаны сценарии использования API (пример запросов и ответов).  
- Нет схемы взаимодействия фронтенда и бэкенда (диаграмма потоков данных).  
- Требуется дополнить документацию по тестированию и отладке.
